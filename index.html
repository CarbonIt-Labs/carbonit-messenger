Why CarbonIt?

Our Vision and Mission

At CarbonIt we treat **privacy as a design decision, not an afterthought**. Our platform combines cutting-edge cryptography with minimal, distraction-free design so users keep ownership of their conversations, even as compute power evolves. Built with future-resistant key exchange and a commitment to privacy-by-default, CarbonIt removes the tradeoffs between security and usability. We believe messages should be private by default, transparent by design, and effortless to use. Whether you’re a developer auditing protocols or someone who simply values quiet, secure chat. Join us in rebuilding communication from the ground up: elegant, verifiable, and uncompromisingly private.

The CarbonIt Genesis Story

The journey of CarbonIt began with a simple, profound challenge: to build a messaging platform resistant to the computational power of future quantum computers while still feeling effortless. Our small team of cryptographers and engineers iterated across prototypes, moving from theoretical key-exchange models to hands-on node tests on localhost and private networks. We prioritized a modular architecture: a hardened node engine for secure peer-to-peer transport, a compact symmetric layer for message throughput, and a minimal client that surfaces only what users need. Every release included automated tests for handshake correctness, offline signing checksums, and human-led audits of any cryptographic glue code. The product you see now is the interface — a privacy-first front door to a protocol stack that remains private until it’s production-ready. We focused on verifiability, reproducibility, and developer-friendly tooling so that, when CarbonIt opens, other engineers can inspect, run, and contribute safely.

Key Features

True Peer-to-Peer (P2P)

Communicates directly between two users — **no central server required** for message routing.

End-to-End Encryption (E2EE)

Messages are encrypted using per-session derived chat keys, ensuring only the sender and receiver can read them.

Automatic Key Rotation

Chat keys are rotated every **10 minutes or 25 messages** to guarantee forward secrecy.

Offline Queueing

If a peer is temporarily offline, messages are **securely queued** and automatically retried upon connection restore.

Message Acknowledgement (ACK)

Each message includes **timestamps and delivery confirmations** for complete sending transparency.

Secure Key Derivation

Session keys are derived from a unique **private–public key exchange** (like Signal-style Diffie–Hellman).

Security Design Layers

Layer Description
Identity Each user generates a deterministic private–public keypair from their username and password.
Handshake A shared session key is derived between peers using their public IDs (Signal-style ECDH).
Encryption Messages use Fernet (AES-128 in CBC with HMAC) encryption for robust security.
Rotation Chat keys are rotated automatically every 10 minutes or 25 messages for forward secrecy.
Integrity Each message includes SHA-256 ID, timestamp (stap), and delivery acknowledgment (ack).

⚙️ How to Run

Follow these steps to clone the repository and run a node:

1. Clone the Repository:

git clone https://github.com/CarbonIt-Labs/carbonit-messenger.git
cd carbonit-messenger/network

2. Run the Node:

Use the batch file (Windows) or the Python script directly:

python carbonit_node.py <peer_public_id_or_None> <peer_ip> <peer_port>

Example to connect to another node ('sam') at 127.0.0.1:5050:

python carbonit_node.py sam 127.0.0.1 5050
